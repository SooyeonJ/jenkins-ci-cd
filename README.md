# Jenkins CI/CD Pipeline 구축

형상관리 도구인 **GitLab**을 사용하여 소스 코드 변경 이력을 관리하고,  
**Jenkins 기반 CI/CD 파이프라인을 구축**하여  
애플리케이션 빌드(WAR 패키징)부터 배포까지의 과정을 자동화함

본 파이프라인은 **CI 단계와 CD 단계를 명확히 분리**하여 구성되었으며,  
CI에서는 빌드 산출물 생성에 집중하고,  
CD에서는 이미 생성된 산출물을 안정적으로 배포하는 구조로 설계

---

## 목표

수동 빌드·배포 과정에서 발생할 수 있는  
인적 오류 및 반복 작업을 최소화하고,  
배포 일관성과 운영 안정성을 확보하는 것을 목표로 함

- 형상관리 기반 소스 코드 관리 체계 확립
- CI/CD 파이프라인 자동화
- WAR 기반 애플리케이션 배포 표준화
- 배포 안정성 및 재현성 확보

---

## 전체 구성 개요

| 구분 | 내용 |
|----|----|
| 형상관리 | GitLab |
| CI 도구 | Jenkins (CI) |
| 빌드 산출물 | WAR |
| 아티팩트 저장소 | Nexus |
| CD 도구 | Jenkins (CD) |
| 배포 대상 | WAS 서버 |

---

## CI 단계 (Continuous Integration)

CI 단계에서는 **소스 코드 빌드 및 패키징**을 담당

### CI 흐름

1. 개발자가 GitLab 저장소에 코드 Push
2. 기관 내 Jenkins(CI)가 GitLab 저장소로부터 소스 코드 Fetch
3. Maven 기반 빌드 수행  
   - 컴파일  
   - 아카이브  
   - WAR 파일 패키징
4. 빌드 완료된 WAR 파일을 Nexus에 업로드

### CI 역할 정리

- 소스 코드 컴파일
- 빌드 산출물(WAR) 생성
- 배포 가능한 아티팩트 단일화
- 빌드 결과의 중앙 저장소(Nexus) 관리

> CI 단계에서만 컴파일 및 패키징 수행  
> CD 단계에서는 소스 코드 및 빌드 과정 미포함

---

## CD 단계 (Continuous Deployment)

CD 단계에서는 **이미 생성된 WAR 파일을 배포 대상 서버에 전달 및 배포**함.

### CD 흐름

1. Jenkins(CD)가 Nexus로부터 WAR 파일 다운로드
2. 다운로드한 WAR 파일을 대상 WAS 서버로 배포
3. 애플리케이션 기동 및 서비스 반영

### CD 역할 정리

- 소스 코드 접근 없음
- 컴파일 및 빌드 수행 없음
- Nexus에 저장된 WAR 파일 기반 배포 수행
- 배포 안정성 및 재현성 확보

> CD Jenkins는 배포 전용 역할 수행  
> 빌드 환경과 배포 환경을 명확히 분리

---

## CI / CD 분리 설계 이유

| 항목 | 설명 |
|----|----|
| 책임 분리 | 빌드(CI)와 배포(CD) 역할 분리 |
| 안정성 | 빌드 실패가 배포 환경에 영향 미치지 않음 |
| 재현성 | 동일 WAR 파일로 다수 환경 배포 가능 |
| 확장성 | 배포 대상 서버 증가 시 CD 확장 용이 |
